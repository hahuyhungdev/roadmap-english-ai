### 1. In your recent work, where do you think compute waste happens most?

**Common ideas:** repeated API calls, heavy media, unnecessary polling, oversized models, background jobs, weak caching.
**Useful language:** "repeated work", "hidden waste", "at scale", "resource-heavy default."
**Pattern to apply:** waste source -> why it happens -> practical fix.

**Version 1 (Clear B2):**
Compute waste often happens in repeated work that nobody questions. For example, repeated requests, oversized images, and unnecessary background jobs can all add cost. Each one looks small, but at scale the waste becomes real.

**Version 2 (Real situation):**
In frontend work, I notice waste in heavy assets, extra libraries, and refetching data more often than necessary. The product still works, so the waste is easy to ignore. But it affects performance, cost, and environmental impact.

**Version 3 (2026 context):**
In 2026, I would also watch AI-related waste. Teams may call large models for simple tasks, keep long context when it is not needed, or regenerate outputs too often. Sustainable engineering now includes asking whether the compute level matches the value of the task.

### 2. What simple sustainable default can be added to one product you use?

**Common ideas:** lower default video quality, lazy loading, fewer autoplay items, compressed media, opt-in heavy features.
**Useful language:** "sensible default", "user control", "lower-energy baseline", "not removing choice."
**Pattern to apply:** default -> user impact -> sustainability gain.

**Version 1 (Clear B2):**
A simple sustainable default is loading lighter media first. Users can still choose higher quality, but the product should not use the heaviest option automatically. Defaults shape behavior, so they matter.

**Version 2 (Real situation):**
For a video app, I would start with standard quality and make higher quality easy to choose. That reduces waste for users who do not need maximum quality every time. It also keeps the experience flexible.

**Version 3 (2026 context):**
For AI products, a sustainable default could be using a smaller model for simple requests and escalating only when needed. The user still gets help, but the system avoids using expensive compute for low-value tasks.

### 3. How would you explain a green trade-off to PM or leadership without sounding symbolic?

**Common ideas:** connect to cost, performance, user control, reliability, measurable impact; avoid moral slogans only.
**Useful language:** "operational cost", "resource efficiency", "user impact", "measurable trade-off."
**Pattern to apply:** recommendation -> business/product impact -> trade-off.

**Version 1 (Clear B2):**
I would explain it through cost, performance, and user impact. For example, a lighter default can reduce infrastructure cost and improve load time, while still allowing users to choose a heavier option.

**Version 2 (Real situation):**
I would avoid saying only "this is greener." I would say what changes in the product: fewer requests, smaller media, lower compute hours, or better cache behavior. That makes the green argument more practical.

**Version 3 (2026 context):**
In 2026, I would also include AI compute in the explanation. If a smaller model handles 80% of simple cases, we can reduce cost and environmental load while keeping quality gates for complex cases. That sounds like engineering judgment, not symbolism.

### 4. Which metric can show sustainability progress without heavy new tooling?

**Common ideas:** transferred data, request count, page weight, compute hours, job frequency, cache hit rate, model calls.
**Useful language:** "proxy metric", "lighter over time", "not perfect but useful", "operational signal."
**Pattern to apply:** metric -> why it helps -> limitation.

**Version 1 (Clear B2):**
A simple metric is transferred data for a key user flow. It is not a perfect emissions metric, but it shows whether the product is becoming lighter or heavier. That is a useful first signal.

**Version 2 (Real situation):**
For frontend work, I would track bundle size, image weight, and request count. For backend work, I would track compute hours, job frequency, or cache hit rate. These metrics are already close to normal engineering work.

**Version 3 (2026 context):**
For AI features, I would track model calls, average context size, and fallback rate to larger models. These are practical proxy metrics. They help teams reduce waste without waiting for a full carbon accounting system.

### 5. What are the benefits and limits of carbon-aware scheduling in real teams?

**Common ideas:** useful for flexible batch jobs; limited by reliability, deadlines, user-facing work, complexity.
**Useful language:** "flexible workload", "cleaner time window", "not a universal answer", "timing constraint."
**Pattern to apply:** where it works -> where it fails -> balanced view.

**Version 1 (Clear B2):**
Carbon-aware scheduling can help when workloads are flexible. Some batch jobs or reports can run at lower-impact times. The limit is that many user-facing tasks cannot wait.

**Version 2 (Real situation):**
I would use it for background processing, not for critical user actions. If a payment or production incident needs immediate response, reliability comes first. Sustainability has to respect the system's purpose.

**Version 3 (2026 context):**
As cloud platforms improve sustainability tooling, carbon-aware scheduling may become easier. But real teams still need to consider deadlines, regions, data rules, and operational complexity. It is a useful tool, not a complete strategy.

### 6. How do sustainability priorities differ between beginner and experienced engineers?

**Common ideas:** beginners improve local waste; experienced engineers influence architecture, defaults, infrastructure, culture.
**Useful language:** "local optimization", "systemic leverage", "scope of responsibility", "engineering principle."
**Pattern to apply:** beginner scope -> senior scope -> shared responsibility.

**Version 1 (Clear B2):**
Beginner engineers can focus on local waste, like image size, repeated requests, or unnecessary libraries. Experienced engineers usually have more influence over architecture, infrastructure, and product defaults.

**Version 2 (Real situation):**
Both levels can contribute. A junior engineer can notice waste in one flow. A senior engineer can ask whether the whole system is designed to avoid repeated compute and unnecessary data movement.

**Version 3 (2026 context):**
With AI systems, beginners may learn to use smaller tools and verify outputs efficiently. Experienced engineers need to set standards for model choice, caching, evaluation, and cost control. Sustainability becomes part of technical leadership.

### 7. When should reliability be prioritized over emission reduction, and why?

**Common ideas:** critical flows, safety, payments, healthcare, incidents, legal obligations; reduce waste around reliability boundary.
**Useful language:** "protected constraint", "service trust", "critical flow", "optimize around it."
**Pattern to apply:** critical context -> reliability first -> greener alternatives around it.

**Version 1 (Clear B2):**
Reliability should come first when users depend on the system in a critical way. If failure creates serious harm, we should not reduce reliability just to lower emissions.

**Version 2 (Real situation):**
For payments, healthcare, or production incidents, stable service matters most. But sustainability can still guide choices around caching, efficient architecture, and waste reduction once reliability is protected.

**Version 3 (2026 context):**
In AI-enabled systems, reliability also includes safe and accurate behavior. I would not choose a lower-compute option if it creates unacceptable risk. The better approach is to define reliability as a boundary, then optimize resource use inside that boundary.

**Common ideas:** be honest, not all green changes save money immediately; compare short-term cost and long-term value.
**Useful language:** "investment decision", "upfront cost", "long-term value", "honest trade-off."
**Pattern to apply:** acknowledge cost -> compare value -> decide.

**Version 1 (Clear B2):**
If greener options are more expensive now, I would say that honestly. Sustainability is not always a free win. The team should compare short-term cost with long-term value and impact.

**Version 2 (Real situation):**
Some improvements reduce cost quickly, like fewer requests or smaller assets. Others may require upfront investment, like infrastructure changes. I would not oversell the benefit; I would present the trade-off clearly.

**Version 3 (2026 context):**
In 2026, leadership may be careful with spending, so green proposals need stronger business logic. I would show cost, risk, user impact, and future savings where possible. Honest numbers build more trust than optimistic slogans.

### 9. You said small optimizations matter. How do you prove they are not symbolic only?

**Common ideas:** show frequency, scale, baseline, before/after metric, high-volume flow, cumulative effect.
**Useful language:** "volume context", "repeated at scale", "before-and-after", "not only percentage."
**Pattern to apply:** small change -> scale/frequency -> measured impact.

**Version 1 (Clear B2):**
I would prove it by showing how often the behavior happens. A small reduction in a high-volume flow can matter. Without frequency or scale, the claim may sound symbolic.

**Version 2 (Real situation):**
For example, reducing one image by 20% is not impressive by itself. But if that image loads millions of times, the impact becomes meaningful. The proof needs both the optimization and the usage context.

**Version 3 (2026 context):**
For AI features, a small prompt or routing optimization can save a lot if it applies to many requests. I would show the baseline, the new metric, and the volume. That turns sustainability from a slogan into measurable engineering.

### 10. If users prefer high-quality video by default, should teams still use low-energy defaults?

**Common ideas:** maybe yes if acceptable quality and easy opt-up; avoid punishing users; use adaptive defaults.
**Useful language:** "perceived quality", "easy upgrade", "adaptive default", "resource efficiency."
**Pattern to apply:** user preference -> design compromise -> reason.

**Version 1 (Clear B2):**
Teams can use lower-energy defaults if the experience remains acceptable and users can upgrade easily. The default should reduce waste without making users feel punished.

**Version 2 (Real situation):**
I would choose a sensible default, not the lowest possible quality. If users are on a small screen or weak network, lower quality may be fine. If they want higher quality, the product should make it simple.

**Version 3 (2026 context):**
Adaptive defaults are probably better than one fixed rule. The product can consider device, network, user setting, and content type. Sustainability should work with user experience, not against it.

**Useful language:** "mature engineering judgment", "not replacing speed", "responsible system design", "wider consequences."
