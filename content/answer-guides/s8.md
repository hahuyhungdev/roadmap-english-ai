### 1. How do you run RCA step by step after a production failure?

**Common ideas:** user symptom, timeline, recent changes, hypothesis testing, confirmed cause.
**Useful language:** "I start with the user impact...", "The timeline shows...", "I only call it root cause when..."
**Pattern to apply:** Symptom -> timeline -> evidence -> hypothesis -> confirmed cause.

**Version 1 (Clear B2):**
I start with the exact user problem, not just the error message. Then I check when it started, which release or config changed, and which user flow was affected. I build a simple timeline from alerts, logs, deploy history, and user reports. After that, I test the strongest hypotheses and rule out weaker ones. I only call it root cause when the evidence explains both the failure and the timing.

**Version 2 (Expanded natural answer):**
In a real incident, I start from the customer journey. Are users unable to log in, complete payment, upload files, or receive AI responses? Then I freeze the timeline: first failure, recent change, alert time, and support reports.

After that, I compare successful and failed sessions in the same time window. For modern systems, I also check provider status, queues, feature flags, and AI-related layers if needed. The goal is not a dramatic story. It is a practical explanation of what failed, who was affected, and what changed after that.

### 2. Which logs, metrics, and traces do you check first, and why?

**Common ideas:** user flow metric, error rate, latency, request volume, traces, provider status.
**Useful language:** "I start closest to the user action...", "The metric depends on the flow...", "Traces help when..."
**Pattern to apply:** User action -> product metric -> technical signals -> dependency check.

**Version 1 (Clear B2):**
I start with signals closest to the user action: error rate, latency, request volume, and success rate of the affected flow. Then I check application logs for repeated errors and status codes. If the request crosses several services, I use traces to see where it slowed down or failed. That helps me separate frontend, backend, database, and provider problems.

**Version 2 (Expanded natural answer):**
If checkout is failing, I do not start with random server logs. I first check checkout attempts, payment success rate, failed API calls, and support tickets. If login is broken, I check auth provider errors, session creation, cookie behavior, and browser errors.

The metric depends on the user flow. For modern systems, I would check product analytics, frontend monitoring, backend metrics, traces, and provider status. The point is to follow the request path, not guess from habit.

### 3. How do you quantify impact window, affected transactions, and revenue risk?

**Common ideas:** impact window, failed actions, baseline, delayed revenue, confidence level.
**Useful language:** "The impact window was...", "Revenue at risk means...", "This is an estimate, not a final number."
**Pattern to apply:** Window -> affected actions -> baseline comparison -> confidence.

**Version 1 (Clear B2):**
I define the impact window from the first clear failure until the system returns to normal. Then I count total attempts, successful actions, failed actions, and affected segments during that period. For revenue risk, I compare missing successful transactions with a normal baseline. I give a rough but honest estimate and update it when better data arrives.

**Version 2 (Expanded natural answer):**
I like impact numbers people can picture. For example: "From 10:12 to 10:48, around 2,400 checkout attempts happened, 620 failed, and the revenue at risk was about $18,000 based on normal order value." That is clearer than saying checkout had issues.

I would also separate lost revenue from delayed revenue because some users may retry later. For AI features, the impact may be trust risk or support load, not immediate lost revenue. If the data is incomplete, I state the confidence level.

### 4. How do you write an RCA summary that is honest without creating blame?

**Common ideas:** facts, user impact, confirmed cause, fix, prevention, no personal blame.
**Useful language:** "The summary should separate facts from blame...", "The safeguard missing was...", "The follow-up is..."
**Pattern to apply:** User impact -> cause -> fix -> prevention -> owner.

**Version 1 (Clear B2):**
I separate facts from blame. A good RCA summary says what users experienced, when it happened, what evidence confirmed the cause, what we fixed, and what we will change next. I avoid naming one person as the problem. The focus should be the missing safeguard or system weakness.

**Version 2 (Expanded natural answer):**
I would write something like: "For 36 minutes, some users could not complete payment because a validation change rejected valid orders. We rolled back the change and added a contract test for that payment path." That is honest, but not personal.

For AI or provider incidents, I would avoid vague blame like "the AI broke" or "the vendor failed." I would describe the chain: provider latency increased, our queue did not protect users, and fallback messaging was missing. That gives the team something concrete to improve.

### 5. When is a fast mitigation acceptable, and when does it create long-term risk?

**Common ideas:** reversible mitigation, reduced impact, owner, deadline, hidden temporary state.
**Useful language:** "Fast mitigation is acceptable when...", "The risk appears when...", "Temporary needs an owner."
**Pattern to apply:** Mitigation -> why acceptable -> hidden risk -> follow-up rule.

**Version 1 (Clear B2):**
A fast mitigation is acceptable when it reduces user impact quickly and is easy to reverse. Rolling back a release or disabling a feature flag can be reasonable. It becomes risky when the team treats the temporary state as normal. A mitigation needs an owner, a follow-up task, and a deadline.

**Version 2 (Expanded natural answer):**
If a recommendation module breaks the homepage, I would rather hide that module than keep the whole page unstable. That is a good mitigation because users can still use the main product. But if we leave it disabled for weeks without understanding the failure, we lose product value and learn nothing.

A mitigation is not a final fix. It reduces harm while diagnosis continues. The trade-off is speed versus learning. Temporary decisions should be visible and owned.

### 6. How do AI features and third-party platforms change incident investigation?

**Common ideas:** external dependencies, provider status, prompt changes, rate limits, fallback behavior.
**Useful language:** "RCA becomes less local...", "I would not immediately blame...", "The failure layer matters..."
**Pattern to apply:** User symptom -> internal checks -> provider checks -> fallback review.

**Version 1 (Clear B2):**
They make RCA less local. The issue may come from a model provider, auth service, payment gateway, CDN rule, queue, prompt change, or retrieval step. So I need to check internal signals and external dependencies before naming the cause. I also need to separate what users experienced from which layer failed.

**Version 2 (Expanded natural answer):**
If an AI assistant becomes slow, I would not immediately blame the model provider. I would check prompt size, retrieval quality, queue depth, network latency, provider status, and rate limits. The user only sees that the answer is slow, but the cause may sit in a different layer.

With external platforms, I also think about fallback and policy risk. Switching providers may protect speed but create data or compliance concerns. Incident response needs technical evidence and product judgment.

### 7. What alternative cause should you rule out before confirming root cause?

**Common ideas:** provider issue, frontend bug, traffic spike, cache, deployment, config.
**Useful language:** "I would rule out...", "The evidence showed...", "That made the RCA stronger."
**Pattern to apply:** Claimed cause -> alternative cause -> evidence -> conclusion.

**Version 1 (Clear B2):**
I should rule out the most plausible alternative causes. For example, if checkout fails, I would check provider status, frontend errors, backend response shape, recent deploys, and traffic spikes. If I can show why those are not the cause, the RCA becomes stronger. It proves the conclusion is based on evidence, not confidence.

**Version 2 (Expanded natural answer):**
If the first guess is a payment provider outage, I would check provider status and response logs. If the provider is normal and our logs show malformed request data after a backend change, then the provider is not the root cause.

For an AI assistant issue, I would not stop at "the model was slow." I would check prompt length, retrieval, queue backlog, rate limits, and our fallback behavior. Clear RCA needs ruled-out paths, not just a final label.

### 8. If business impact looked low at first but grew later, what did the analysis miss?

**Common ideas:** delayed impact, indirect effects, customer segment, retries, support load.
**Useful language:** "The first estimate was too narrow...", "It missed delayed effects...", "I would update the impact model..."
**Pattern to apply:** Initial estimate -> missed impact -> new data -> updated model.

**Version 1 (Clear B2):**
The first estimate probably missed delayed or indirect effects. We may count only failed requests at first, then later see abandoned sessions, retries, refund requests, support tickets, or churn risk. The estimate can also be too narrow if it ignores which customer segment was affected. Impact is not only what failed immediately.

**Version 2 (Expanded natural answer):**
Maybe the technical failure rate was only 3 percent, so it looked small. But if those users were all new customers trying to subscribe, the business impact could be much bigger. The same failure rate can mean different things depending on who was affected.

Impact can also grow through trust and support effects. If an AI feature gives bad answers to enterprise users, direct error count may look low, but account risk can be high. I would update the model and include second-order effects.

### 9. How do you avoid blaming the wrong team when evidence is incomplete?

**Common ideas:** facts versus interpretation, shared timeline, weak proof, probable cause, contributing factors.
**Useful language:** "I separate facts from interpretation...", "The evidence supports...", "Contributed to the issue..."
**Pattern to apply:** Known facts -> unknowns -> hypotheses -> fair wording.

**Version 1 (Clear B2):**
I separate facts from interpretation. I write down what we know, what we do not know, and which evidence supports each hypothesis. If proof is weak, I call it a probable cause, not a confirmed cause. The RCA should follow evidence and shared timelines, not team politics.

**Version 2 (Expanded natural answer):**
When frontend, backend, and platform teams disagree, I put the timeline and data in one place first. For example, if the UI showed an error, I still check whether the API response was already invalid before frontend rendered it. That avoids emotional blame.

With vendors and AI providers, wording matters. I might say, "Provider latency contributed to the issue, but our missing fallback increased the impact." That is fair. It names responsibility without turning the RCA into personal blame.

### 10. What makes an RCA useful after the incident is over?

**Common ideas:** prevention, owners, deadlines, monitoring, tests, fallback, changed behavior.
**Useful language:** "An RCA is useful if...", "The follow-up should include...", "If nothing changes..."
**Pattern to apply:** Explanation -> impact -> action items -> changed system.

**Version 1 (Clear B2):**
An RCA is useful if it changes future behavior. It should explain what happened, why it happened, how users were affected, and what the team will do differently. The follow-up should include owners, deadlines, tests, monitoring, or fallback changes. If nothing changes afterward, the RCA is mostly paperwork.

**Version 2 (Expanded natural answer):**
For me, a useful RCA has concrete follow-up. After a checkout incident, that might mean adding a contract test, improving a feature flag checklist, and creating an alert for payment failure rate. Product should also understand the impact clearly.

In modern systems, prevention is not always "remove the bug forever." Sometimes it means detecting degradation earlier, reducing blast radius, giving users a fallback, and making ownership visible. The RCA is useful only when the next similar incident is less painful.
