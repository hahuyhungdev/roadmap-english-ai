### 1. How do you run RCA step by step after a production failure?

**Version 1 (Clear B2):**
After a production failure, I first define the exact user problem, not just the error message. Then I check when it started, which release or config changed near that time, and which user flow is affected. I build a simple timeline from alerts, logs, deploy history, and user reports. After that, I test one or two likely causes and rule out weaker ones. I only call it the root cause when the evidence explains both the failure and the timing.

**Version 2 (Product reality):**
In a real product, I start with the customer journey. For example, are users unable to log in, complete payment, upload files, or receive AI responses? Then I compare successful and failed sessions in the same time window. I also check whether the issue is limited to one region, one plan, one browser, or one third-party provider. This helps me avoid a vague RCA like "the API failed." A useful RCA should say which part failed, why it failed, who was affected, and what we changed.

**Version 3 (2026 context):**
In 2026, I would also check AI and platform dependencies early because many products now rely on model APIs, queues, edge caching, and third-party auth. My steps are: confirm the symptom, freeze the timeline, check recent changes, compare observability data, test hypotheses, and document the confirmed cause. If an AI feature is involved, I separate model latency, prompt changes, rate limits, and backend bugs. That keeps the analysis practical instead of blaming "AI instability" too quickly.

### 2. Which logs, metrics, and traces do you check first, and why?

**Version 1 (Clear B2):**
I usually start with the signals closest to the user action: error rate, latency, request volume, and the success rate of the affected flow. Then I check application logs for repeated errors and status codes. If the system has multiple services, I use traces to see where the request slowed down or failed. I choose these first because they quickly show whether the issue is frontend, backend, database, or integration.

**Version 2 (Product reality):**
If checkout is failing, I do not start by reading random server logs. I first check checkout attempts, payment success rate, failed API calls, and support tickets. If login is broken, I check auth provider errors, session creation, cookie behavior, and browser errors. The metric depends on the flow. Good RCA starts from the business action, then moves into technical detail.

**Version 3 (2026 context):**
For modern systems, I check four layers: product analytics, frontend monitoring, backend metrics, and provider status. A 2026 incident may come from a model provider timeout, a CDN rule, a bot-protection change, or a queue backlog, not only our own code. Traces help when one user request crosses several services. I also check rate-limit dashboards because traffic spikes from automation or AI agents can create strange failure patterns.

### 3. How do you quantify impact window, affected transactions, and revenue risk?

**Version 1 (Clear B2):**
I define the impact window from the first clear failure until the system returns to normal. Then I count total attempts, successful actions, and failed actions during that period. For revenue risk, I compare the missing successful transactions with a normal baseline. I do not pretend the first number is perfect. I give a rough but honest estimate and update it when better data arrives.

**Version 2 (Product reality):**
I like to explain impact with numbers people can picture. For example: "From 10:12 to 10:48, around 2,400 checkout attempts happened, 620 failed, and the estimated revenue at risk was about $18,000 based on our normal order value." That is much clearer than saying "checkout had issues." I also separate lost revenue from delayed revenue because some users may retry later.

**Version 3 (2026 context):**
In 2026, I would include indirect impact too. A failed AI assistant may not lose revenue immediately, but it can increase support tickets, reduce trial conversion, or make enterprise customers lose trust. So I estimate direct failures, delayed actions, user drop-off, and support load. If the data is incomplete, I state the confidence level. A useful impact number is not perfect; it is transparent enough for product and leadership to act.

### 4. How do you write an RCA summary that is honest without creating blame?

**Version 1 (Clear B2):**
I would separate facts from blame. A good RCA summary should say what users experienced, when it happened, what evidence confirmed the cause, what we fixed, and what we will change next. I avoid naming people as the problem. The focus should be system weakness, decision gap, or missing safeguard.

**Version 2 (Product reality):**
I would write something like: "For 36 minutes, some users could not complete payment because a validation change rejected valid orders. We rolled back the change and added a contract test for that payment path." This is honest because it names the failure clearly. But it does not attack the person who made the change. The useful question is what safeguard was missing.

**Version 3 (2026 context):**
For AI or provider-related incidents, I would avoid vague blame like "the AI broke" or "the vendor failed." I would describe the chain: provider latency increased, our queue did not protect users, and fallback messaging was missing. That wording is more useful because it shows shared system responsibility. A strong RCA should make prevention easier, not make teams defensive.

### 5. What are the benefits and limits of deep diagnosis before choosing a fix?

**Version 1 (Clear B2):**
Deep diagnosis helps the team avoid fixing the wrong thing. It shows whether the visible error is the real cause or only a symptom. But during an active incident, time matters. If users are blocked, we may need a safe mitigation first and deeper analysis after the system is stable.

**Version 2 (Product reality):**
The benefit is that the final fix becomes stronger. For example, if payments fail because of a timeout, increasing the timeout may help today, but it may hide a slow database query. The limit is that customers do not care about a beautiful theory while the product is down. My preferred approach is restore the service first, then finish the deeper RCA before closing the incident.

**Version 3 (2026 context):**
Deep diagnosis is even more important when systems depend on many layers: model APIs, cloud queues, feature flags, analytics scripts, and security tools. A shallow fix can move the problem from one layer to another. Still, there is a real limit. If the issue affects live revenue or safety, mitigation comes first. Deep diagnosis should improve prevention, not delay recovery without a clear reason.

### 6. When is a fast mitigation acceptable, and when does it create bigger long-term risk?

**Version 1 (Clear B2):**
A fast mitigation is acceptable when it reduces user impact quickly and is easy to reverse. Rolling back a release, disabling a feature flag, or routing traffic away from one broken service can be reasonable. It becomes risky when the team forgets it is temporary. Then the same weakness stays in the system.

**Version 2 (Product reality):**
If a new recommendation feature breaks the homepage, I would rather hide that module than keep the whole homepage unstable. That is a good mitigation because users can still use the main product. But if we leave the module disabled for weeks without understanding the failure, we lose product value and learn nothing. A mitigation needs an owner, a follow-up task, and a deadline.

**Version 3 (2026 context):**
In 2026 systems, fast mitigation may mean switching model providers, lowering AI response quality, turning off personalization, or adding a queue. That can protect users, but it may also hide cost, privacy, or reliability problems. For example, moving traffic to another provider without checking data policy can create compliance risk. Fast is fine, but it still needs boundaries.

### 7. How do AI features and third-party platforms change the way you investigate incidents?

**Version 1 (Clear B2):**
They make RCA less local. The problem may not be only in our code. It can come from a model provider, auth service, payment gateway, CDN rule, queue, prompt change, or retrieval step. So I need to check both internal signals and external dependencies. I also need to separate what users experienced from which layer actually caused it.

**Version 2 (Product reality):**
For example, if an AI assistant becomes slow, I would not immediately blame the model provider. I would check prompt size, retrieval quality, queue depth, network latency, provider status, and rate limits. The same logic applies to payment or auth incidents. Modern RCA is about the whole user flow, not only one service.

**Version 3 (2026 context):**
In 2026, many products are built on external platforms and AI layers, so RCA needs stronger boundary checks. I would ask: did our system send bad input, did the provider degrade, did traffic hit a rate limit, or did our fallback fail? I also think about data policy before switching providers as a mitigation. The goal is to protect the user experience without creating a new reliability or compliance problem.

### 8. You said your root cause is clear. Which alternative cause did you rule out, and how?

**Version 1 (Clear B2):**
If I say the root cause is clear, I should show what I ruled out. For example, I might rule out a frontend bug by testing the same request through an API client and checking that the backend response is already wrong. I might rule out traffic volume if the error happened at normal load. Ruling out alternatives makes the RCA stronger.

**Version 2 (Product reality):**
In one payment issue, the first guess was a payment provider outage. But the provider status was normal, and our logs showed that we sent malformed request data after a backend change. That ruled out the provider as the main cause. This matters because blaming a vendor is easy, but it does not help if the real issue is our integration code.

**Version 3 (2026 context):**
For an AI assistant issue, I would not stop at "the model was slow." I would check whether the prompt got longer, whether retrieval added too many documents, whether rate limits changed, and whether our queue was backed up. If traces show most delay happened before the model call, then the model provider is not the root cause. This kind of separation is important now because AI systems have many moving parts.

### 9. If business impact looked low at first but grew later, what did your analysis miss?

**Version 1 (Clear B2):**
It probably missed delayed or indirect effects. At first, we may count only failed requests, but later we see abandoned sessions, retries, refund requests, or support tickets. The first estimate can be too narrow. A good analysis should be updated when new impact appears.

**Version 2 (Product reality):**
Maybe the technical failure rate was only 3%, so it looked small. But if those users were all new customers trying to subscribe, the business impact could be much bigger. Or maybe users did not fail immediately; they got stuck, waited, and left. That is why I check funnel drop-off and customer segment, not only server errors.

**Version 3 (2026 context):**
In 2026, impact can grow through trust and automation effects. If an AI feature gives bad answers to enterprise users, the direct error count may look low, but the account risk can be high. If bots or AI agents keep retrying failed actions, the load can also grow after the original bug. So I would revisit the impact model and include second-order effects.

### 10. How do you avoid blaming the wrong team when evidence is incomplete?

**Version 1 (Clear B2):**
I avoid blame by separating facts from interpretation. I write down what we know, what we do not know, and which evidence supports each hypothesis. If the proof is weak, I say it is a probable cause, not a confirmed cause. That is more useful than forcing a clean story too early.

**Version 2 (Product reality):**
When frontend, backend, and platform teams disagree, I put the timeline and data in one place first. For example, if the UI showed an error, I still check whether the API response was already invalid before the frontend rendered it. That helps avoid emotional blame. The RCA should follow the evidence, not team ownership politics.

**Version 3 (2026 context):**
With vendors and AI providers, blame can become even messier. A provider may be slow, but our retries, prompt size, queue, or fallback design may make the user impact worse. So I would say something like, "Provider latency contributed to the issue, but our lack of fallback increased the impact." That wording is fair and gives the team something concrete to improve.

### 11. Which part of RCA is hardest for you to explain in English?

**Version 1 (Clear B2):**
The hardest part is explaining the difference between symptom, trigger, and root cause. In English, I sometimes make that story too long. I try to use a simple order: what users saw, what changed, what evidence we found, and what we changed after that.

**Version 2 (Product reality):**
For me, it is hard to explain contributing factors without sounding like I am making excuses. For example, weak monitoring, unclear ownership, and a rushed release may all contribute to an incident. I need to say them honestly but calmly. Short sentences help: "The code bug caused the failure. Weak monitoring made detection slower."

**Version 3 (2026 context):**
AI-related RCA is especially hard to explain in English because words like hallucination, retrieval, latency, and rate limit can sound abstract. I try to translate them into user impact. Instead of saying "retrieval quality dropped," I might say, "The assistant pulled less relevant documents, so answers became less accurate for support questions." That is clearer and more useful.

### 12. What debugging habit most improved your diagnosis quality?

**Version 1 (Clear B2):**
The habit that helped me most is writing down the timeline while debugging. It stops me from mixing facts with guesses. I can see what happened first and what happened later. That makes my conclusion much more reliable.

**Version 2 (Product reality):**
I improved a lot when I started testing one hypothesis at a time. Before, I jumped between dashboards, logs, and code too quickly. Now I write: "If this cause is true, what evidence should I see?" Then I check that evidence. It makes debugging calmer and reduces random investigation.

**Version 3 (2026 context):**
Recently, I have found it useful to save evidence links during the incident: dashboard screenshots, trace IDs, release IDs, provider status pages, and support examples. Modern incidents move across many tools, so memory is not enough. Keeping evidence together makes the final RCA faster and makes team discussion less emotional.

### 13. What makes an RCA actually useful after the incident is over?

**Version 1 (Clear B2):**
An RCA is useful if it changes future behavior. It should explain what happened, why it happened, how users were affected, and what the team will do differently. If it only says "we fixed it" without prevention, it is not very useful. The best RCA creates better monitoring, tests, ownership, or rollout rules.

**Version 2 (Product reality):**
For me, a useful RCA has owners and deadlines. For example, after a checkout incident, the follow-up might be adding a contract test, improving a feature flag checklist, and creating an alert for payment failure rate. Product should also understand the impact clearly. If nobody can point to a changed system or process two weeks later, the RCA was mostly paperwork.

**Version 3 (2026 context):**
In 2026, a useful RCA should also improve fallback behavior. Systems depend on AI providers, cloud services, queues, and security tools, so prevention is not always "remove the bug forever." Sometimes it means detecting degradation earlier, reducing blast radius, giving users a clear fallback, and making ownership visible. That is what turns an incident into real learning.
