### 1. Which AI tasks save you the most time in real work?

**Common ideas:** first drafts, tests, docs, code explanation, log summaries.
**Useful language:** "AI helps most when...", "It saves blank-page time", "I still review..."
**Pattern to apply:** Use case -> time saved -> review step -> result.

**Version 1 (Clear B2):**
AI saves me the most time when the task has a clear pattern. I use it for documentation drafts, test ideas, log summaries, and quick code explanations. It helps me avoid blank-page time and understand unfamiliar code faster. But I still review the logic, rewrite the language, and check whether the answer fits our project.

**Version 2 (Expanded natural answer):**
In real work, AI helps me most with first drafts and exploration. For example, I use it to draft test cases, summarize a long error log, prepare release notes, or explain unfamiliar code. It gives me a faster starting point.

But I do not treat that output as final. If the answer is too generic, I rewrite it. If it mentions an API or pattern I do not know, I check the source. The value is not that AI finishes the work alone. The value is that it helps me move faster while I still own correctness and context.

### 2. Where do you avoid AI completely, and why?

**Common ideas:** secrets, customer data, payment, auth, security, private company code.
**Useful language:** "I avoid AI when...", "The cost of a mistake is too high", "The safer option is..."
**Pattern to apply:** High-risk area -> reason -> safer alternative.

**Version 1 (Clear B2):**
I avoid AI when the work involves sensitive data, secrets, customer information, or security-critical logic. I also avoid pasting raw logs into external tools. These areas need strong human control because the cost of a mistake is high. AI can help with general ideas, but privacy and security boundaries come first.

**Version 2 (Expanded natural answer):**
I avoid AI when the task involves private user data, API keys, payment logic, or authentication decisions. Even if AI gives a good-looking answer, I cannot trust it without deep review. I also avoid sharing company code if the tool is not approved by the team.

For example, I would not paste raw customer logs, internal incidents, secrets, or private business data into an external AI tool. If AI is useful, I would anonymize the context or use an approved internal workflow. The trade-off is that it takes a little more time, but it protects users and the company.

### 3. What should be in a practical verification checklist?

**Common ideas:** requirements, edge cases, tests, security, accessibility, local patterns.
**Useful language:** "My checklist is simple", "Before using it, I check...", "Do I understand it well enough to own it?"
**Pattern to apply:** Requirement fit -> edge cases -> tests -> risk -> ownership.

**Version 1 (Clear B2):**
My checklist starts with the requirement. Then I check edge cases, tests, security risk, accessibility, and local code patterns. For frontend work, I also check loading, empty, error, and success states. If AI generated the code, I ask whether I understand it well enough to own it. If I cannot explain it, I should not merge it.

**Version 2 (Expanded natural answer):**
I would keep the checklist short enough to use every day. First, does the output match the requirement? Second, did it handle edge cases, not only the happy path? Third, do tests pass? Fourth, does it follow our existing patterns?

For AI-assisted work, I would add source checks and risk level. If AI mentions an API, I verify it against official docs or our codebase. If the change touches auth, payment, privacy, or user-facing AI behavior, I require deeper review. Verification should match risk, not just code size.

### 4. How do you keep accountability clear when AI contributes to a change?

**Common ideas:** ownership, PR transparency, verification evidence, reviewer focus.
**Useful language:** "AI helped draft...", "I verified it by...", "I still own what I merge."
**Pattern to apply:** AI contribution -> verification -> owner -> reviewer signal.

**Version 1 (Clear B2):**
Accountability stays with the engineer who submits the change. If AI helped write code, tests, or documentation, I still need to understand and verify the final result. I would not say "AI wrote it" as an excuse. A simple rule works well: if I merge it, I own it.

**Version 2 (Expanded natural answer):**
In a pull request, I would be transparent about AI help when it matters. For example, I might say, "AI drafted the first test cases, and I verified them against the payment error scenarios." That helps reviewers know where to focus.

AI can contribute to the workflow, but it is not the final decision-maker. The engineer still owns correctness, context, and safety. This matters because AI can increase output faster than review capacity. Clear accountability keeps AI useful without weakening responsibility.

### 5. What are the benefits and limits of AI-generated code in team workflows?

**Common ideas:** boilerplate speed, first draft, system context, code review, ownership.
**Useful language:** "The benefit is...", "The limit is...", "I treat it like..."
**Pattern to apply:** Benefit -> where useful -> where risky -> review rule.

**Version 1 (Clear B2):**
AI-generated code can save time on small and clear tasks. It is useful for boilerplate, test drafts, utility functions, and simple refactors. The limit is that it may not understand the full system or business rule. So I treat AI code as a draft. It still needs review, tests, and ownership.

**Version 2 (Expanded natural answer):**
AI-generated code is helpful when the task is clear and low risk. It can reduce repetitive work and help the team move faster. But it becomes risky when the task needs deep product context. Sometimes the code looks clean but does not match our data contract, accessibility rule, or privacy boundary.

I would treat AI-generated code like code from a new teammate. It may be useful, but it needs review and explanation before it becomes trusted. If nobody understands why the code works, the speed gain is not real. The team still needs ownership.

### 6. How does AI usage differ between beginner and experienced engineers?

**Common ideas:** shortcut vs thinking partner, prompt quality, review quality, skill growth.
**Useful language:** "The tool is the same, but...", "Experienced engineers...", "Beginners should..."
**Pattern to apply:** Beginner risk -> experienced behavior -> learning rule.

**Version 1 (Clear B2):**
Beginners may use AI mainly to get answers quickly. Experienced engineers use AI to explore options, challenge ideas, and speed up drafts. The difference is review quality and ownership. Experienced engineers can notice when the answer is wrong, incomplete, or too generic. Beginners should explain AI-generated code before using it.

**Version 2 (Expanded natural answer):**
For beginners, AI can reduce fear and explain concepts, which is useful. But it can also become a shortcut. A beginner may copy code that works without understanding the trade-offs or hidden bugs.

Experienced engineers use AI differently. They give more context, ask better questions, and review the answer carefully. They also know when to ignore the suggestion. The difference is not access to the tool; it is judgment. A good team should teach juniors to use AI in a way that supports learning, not replaces reasoning.

### 7. When should quality be prioritized over speed in AI-assisted delivery?

**Common ideas:** critical flow, money, security, privacy, trust, architecture.
**Useful language:** "Quality should win when...", "The cost of a mistake is high", "Speed does not matter if..."
**Pattern to apply:** Risk level -> why speed is not enough -> quality controls.

**Version 1 (Clear B2):**
Quality should win when the feature affects money, security, privacy, trust, or important user data. In those areas, speed is not enough. AI can help with drafts, but we need careful review, tests, and smaller changes. A fast bug in a critical flow is still a bad outcome.

**Version 2 (Expanded natural answer):**
I would prioritize quality when the cost of a mistake is high: checkout, authentication, payment, privacy settings, or anything that affects user trust. AI may help us write faster, but if the result breaks production, the speed does not matter.

For low-risk tasks like documentation drafts or simple test ideas, speed is fine. For high-risk tasks, I want smaller changes, stronger review, and clear ownership. The trade-off is slower delivery, but it protects users and the company. AI changes drafting speed, not the responsibility for safety.

### 8. How do you prove that AI improved speed without lowering quality?

**Common ideas:** defect rate, review rework, tests, CI failures, production bugs.
**Useful language:** "Speed is easy to feel, but quality needs evidence", "I would compare...", "The whole delivery cycle matters."
**Pattern to apply:** Speed claim -> quality metrics -> before/after -> conclusion.

**Version 1 (Clear B2):**
I would prove it with tests, review results, QA feedback, and production metrics. If we move faster but bugs increase, quality dropped. I would compare defect rate before and after using AI. I would also check whether reviewers spend more time fixing AI mistakes. Speed only counts if the whole delivery cycle improves.

**Version 2 (Expanded natural answer):**
Speed is easy to feel, but quality needs evidence. I would check post-merge bugs, failed CI checks, review rework, QA feedback, and support tickets. If AI helps us draft faster and escaped bugs stay the same or decrease, that is a good sign.

But if reviewers spend more time correcting AI mistakes, the speed gain may be fake. We need to measure the whole delivery cycle, not only first-draft time. The real question is whether AI helps us deliver faster with the same or better quality.

### 9. What should a team do if reviewers are overloaded by AI-generated changes?

**Common ideas:** review capacity, smaller changes, automated checks, risk-based review.
**Useful language:** "Manual review works only when...", "I would reduce...", "AI adoption should match..."
**Pattern to apply:** Review bottleneck -> risk -> process adjustment.

**Version 1 (Clear B2):**
If reviewers are overloaded, the team should slow down and reduce scope. Manual review only works when reviewers have enough attention. We can add automated checks, require tests, and keep AI-generated changes smaller. For risky areas, I would limit AI usage until review capacity improves.

**Version 2 (Expanded natural answer):**
When reviewers are overloaded, AI-generated code becomes more dangerous because mistakes can pass through quickly. I would not solve that by asking reviewers to move faster. I would reduce PR size, require tests before review, and use risk-based rules.

Low-risk docs and simple tests can move faster. Auth, payment, privacy, and user-facing AI behavior need deeper review. The trade-off is less speed in the short term, but more control. AI adoption should match the team's ability to verify the output.

**Common ideas:** draft, verify, privacy, ownership.
**Useful language:** "I use AI for...", "I do not use it for...", "I still own..."
**Pattern to apply:** Use -> boundary -> ownership.

**Version 1 (Clear B2):**
I would say, "I use AI for drafts, exploration, tests, and documentation, but I verify the output before using it. I do not share sensitive data, and I still own anything I merge." That summary is simple and practical.

**Version 2 (Expanded natural answer):**
My rule is: AI can help me start faster, but it cannot replace verification or ownership. I use it for first drafts, code explanation, test ideas, and documentation. I avoid it with secrets, customer data, private logs, and high-risk logic unless the workflow is approved.

If I need to summarize it to a manager, I would say: "AI is useful for speed, but the final responsibility stays with the engineer. The safe workflow is draft, verify, review, then ship."
