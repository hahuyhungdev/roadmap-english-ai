### 1. Which AI tasks save you the most time in real work?

**Common ideas:** first drafts, tests, docs, code explanation, log summaries.
**Useful language:** "AI helps most when...", "It saves blank-page time", "I still review..."
**Pattern to apply:** Use case -> time saved -> review step -> result.

**Version 1 (Clear B2):**
AI saves me the most time on repetitive tasks. I use it for first drafts of documentation, test cases, and quick code explanations. It is also useful when I need to understand unfamiliar code faster. But I still review everything before using it.

**Version 2 (Real project):**
In real work, AI helps me most when the task has a clear pattern. For example, writing a first draft of test cases, summarizing a long error log, or creating documentation from a finished feature. It saves time because I do not start from a blank page. After that, I check the logic and rewrite the language for our team.

**Version 3 (2026 context):**
In 2026, I see AI as a drafting and exploration layer. It can help compare implementation options, generate edge-case ideas, summarize issue reports, or prepare release notes. The value is not that AI finishes the work alone. The value is that it gives me a faster starting point, while I still own correctness, context, and final quality.

### 2. Where do you avoid AI completely, and why?

**Common ideas:** secrets, customer data, payment, auth, security, private company code.
**Useful language:** "I avoid AI when...", "The cost of a mistake is too high", "Accountability stays with..."
**Pattern to apply:** High-risk area -> reason -> safer alternative.

**Version 1 (Clear B2):**
I avoid AI for sensitive data, secrets, and security-critical logic. I also avoid copying AI code directly into production without review. These areas need strong human judgment and context. AI can help explain general ideas, but it should not make the final decision.

**Version 2 (Real project):**
I avoid AI when the task involves private user data, API keys, payment logic, or authentication decisions. Those areas are too sensitive. Even if AI gives a good-looking answer, I cannot trust it without deep review. I also avoid sharing company code if the tool is not approved by the team.

**Version 3 (2026 context):**
My boundary is stricter when data or trust is involved. I would not paste raw customer logs, secrets, internal incidents, or private business data into an external AI tool. If AI is needed, I would anonymize the context or use an approved internal workflow. Speed is not worth a privacy or security incident.

### 3. What should be in a practical verification checklist?

**Common ideas:** requirements, edge cases, tests, security, accessibility, local patterns.
**Useful language:** "My checklist is simple", "Before using it, I check...", "Do I understand it well enough to own it?"
**Pattern to apply:** Requirement fit -> edge cases -> tests -> risk -> ownership.

**Version 1 (Clear B2):**
A verification checklist should include requirements, edge cases, tests, security, and code review. For frontend work, I also check loading, empty, error, and success states. If AI generated the code, I check whether it really matches our project. The checklist should be short enough to use every day.

**Version 2 (Real project):**
My checklist is simple. First, does the output match the requirement? Second, did it handle edge cases, not only the happy path? Third, do tests pass? Fourth, does it follow our existing code style and patterns? For AI-generated code, I also ask myself: do I understand every part well enough to own it?

**Version 3 (2026 context):**
For AI-assisted work, I would add source checks and risk level. If AI mentions an API, I verify it against docs or the local codebase. If the change touches auth, payment, privacy, or AI output quality, I require deeper review. Verification should match the risk, not just the size of the code.

### 4. How do you keep accountability clear when AI contributes to a change?

**Common ideas:** ownership, PR transparency, verification evidence, reviewer focus.
**Useful language:** "AI helped draft...", "I verified it by...", "I still own what I merge."
**Pattern to apply:** AI contribution -> verification -> owner -> reviewer signal.

**Version 1 (Clear B2):**
Accountability should stay with the engineer who submits the change. If AI helped write code, tests, or documentation, I should still understand the final result and verify it. I would not say "AI wrote it" as an excuse. A clear rule is simple: if I merge it, I own it.

**Version 2 (Real project):**
In a real pull request, I would be transparent about AI help when it matters. For example, I might say, "AI drafted the first test cases, and I verified them against the payment error scenarios." That helps reviewers know where to focus. It also shows that AI was part of the workflow, but not the final decision-maker.

**Version 3 (2026 context):**
In 2026, accountability is important because AI can increase output faster than review capacity. I would classify AI-assisted changes by risk. Low-risk docs or simple tests can move faster, but auth, payment, privacy, and user-facing AI behavior need stronger review. The core principle stays the same: AI can contribute, but humans own safety, context, and final approval.

### 5. What are the benefits and limits of AI-generated code in team workflows?

**Common ideas:** boilerplate speed, first draft, system context, code review, ownership.
**Useful language:** "The benefit is...", "The limit is...", "I treat it like..."
**Pattern to apply:** Benefit -> where useful -> where risky -> review rule.

**Version 1 (Clear B2):**
AI-generated code can save time and reduce boilerplate. It can help developers move faster on simple tasks. But the limit is that it may not understand the whole system. The team still needs code review, tests, and ownership.

**Version 2 (Real project):**
AI-generated code is helpful when the task is small and clear, like a utility function, test draft, or simple refactor. But it becomes risky when the task needs deep product context. Sometimes the code looks clean but does not match our business rule. I think AI code should go through the same review process as human code.

**Version 3 (2026 context):**
The benefit is faster first drafts, but the limit is context. AI may not understand local architecture, data contracts, accessibility requirements, or privacy boundaries. I would treat AI-generated code like code from a new teammate: useful, but it needs review, tests, and explanation before it becomes trusted.

### 6. How does AI usage differ for beginner versus experienced engineers?

**Common ideas:** shortcut vs thinking partner, prompt quality, review quality, skill growth.
**Useful language:** "The tool is the same, but...", "Experienced engineers...", "Beginners should..."
**Pattern to apply:** Beginner risk -> experienced behavior -> learning rule.

**Version 1 (Clear B2):**
Beginner engineers may use AI to get answers quickly. Experienced engineers use AI to explore options and speed up drafts. The difference is review quality. Experienced engineers know when the AI answer is wrong or incomplete.

**Version 2 (Real project):**
For beginners, AI can explain concepts and reduce fear. But it can be dangerous if they copy code without understanding it. Experienced engineers ask better questions, give more context, and review the answer carefully. They also know when to ignore the suggestion.

**Version 3 (2026 context):**
The main difference is ownership. A beginner may use AI as a shortcut to get code working. An experienced engineer uses AI as a thinking partner but still owns the final decision. I think teams should teach juniors to explain AI-generated code before using it, so AI supports learning instead of replacing reasoning.

### 7. When should quality be prioritized over speed in AI-assisted delivery?

**Common ideas:** critical flow, money, security, privacy, trust, architecture.
**Useful language:** "Quality should win when...", "The cost of a mistake is high", "Speed does not matter if..."
**Pattern to apply:** Risk level -> why speed is not enough -> quality controls.

**Version 1 (Clear B2):**
Quality should be prioritized when the feature affects users directly, money, security, or important data. In those cases, speed is not enough. AI can help with drafts, but we need careful review. A fast bug in a critical flow can be very expensive.

**Version 2 (Real project):**
I would prioritize quality when the cost of a mistake is high, such as checkout, authentication, payment, privacy settings, or anything that affects user trust. AI may help us write code faster, but if the result breaks production, the speed does not matter. I prefer smaller changes, more tests, and human review.

**Version 3 (2026 context):**
Quality should win when AI-assisted work touches high-risk areas: user data, financial actions, identity, compliance, or long-term architecture. I would also be careful with AI features that generate user-facing answers. Bad output can damage trust even if the system technically works. For low-risk tasks, speed is fine. For high-risk tasks, verification is the real priority.

### 8. You said AI improved speed. How do you prove quality did not drop?

**Common ideas:** defect rate, review rework, tests, CI failures, production bugs.
**Useful language:** "Speed is easy to feel, but quality needs evidence", "I would compare...", "The whole delivery cycle matters."
**Pattern to apply:** Speed claim -> quality metrics -> before/after -> conclusion.

**Version 1 (Clear B2):**
I would prove it with tests, review results, and production metrics. If we move faster but bugs increase, quality dropped. I would compare defect rate before and after using AI. I would also check whether QA found more edge-case issues.

**Version 2 (Real project):**
Speed is easy to feel, but quality needs evidence. I would check post-merge bugs, QA feedback, test coverage, and review comments. If AI helps us write tests faster but escaped bugs stay the same or decrease, that is a good sign. But if reviewers spend more time fixing AI mistakes, the speed gain may be fake.

**Version 3 (2026 context):**
I would track both process and outcome metrics. Process metrics include review rework, failed CI checks, and test coverage. Outcome metrics include production bugs, rollback rate, support tickets, and user complaints. AI may reduce first-draft time but increase review time, so I would evaluate the whole cycle.

### 9. You said manual review is enough. What if reviewers are overloaded?

**Common ideas:** review capacity, smaller changes, automated checks, risk-based review.
**Useful language:** "Manual review is only effective when...", "I would reduce...", "AI adoption should match..."
**Pattern to apply:** Review problem -> risk -> process adjustment.

**Version 1 (Clear B2):**
If reviewers are overloaded, manual review may not be enough. The team should reduce scope, add automated checks, or limit AI-generated changes. Review quality matters more than review speed. Otherwise, mistakes can pass through easily.

**Version 2 (Real project):**
When reviewers are overloaded, the risk goes up because people approve code too quickly. In that situation, I would keep AI-generated changes small and require tests before review. I would also avoid using AI for risky areas until review capacity improves. It is better to slow down than create hidden production bugs.

**Version 3 (2026 context):**
AI can increase the amount of code that needs review, so review capacity becomes a real bottleneck. I would add risk-based review rules: low-risk docs and tests can move faster, but auth, payment, privacy, and AI-output logic need deeper review. AI adoption should match the team's ability to verify the output.
