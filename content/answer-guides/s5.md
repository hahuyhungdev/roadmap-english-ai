**Useful language:** "AI helps most when...", "It saves blank-page time", "I still review..."
**Pattern to apply:** Use case -> time saved -> review step -> result.

**Version 1 (Clear B2):**
AI saves me the most time when the task has a clear pattern. I use it for first drafts of documentation, test ideas, log summaries, and quick code explanations. It helps me avoid blank-page time and understand unfamiliar code faster. But I still review the logic, rewrite the language, and check whether the answer fits our project. **Version 2 (Expanded natural answer):**
In real work, AI helps me most when the task has a clear pattern. For example, I use it to draft test cases, summarize a long error log, create documentation from a finished feature, or explain unfamiliar code. I see AI as a drafting and exploration layer, not as a replacement for engineering judgment. It can compare implementation options, suggest edge cases, summarize issue reports, or prepare release notes. The value is not that AI finishes the work alone. The value is that it gives me a faster starting point while I still own correctness, context, and final quality. That distinction keeps the workflow useful and safe.

### 2. Where do you avoid AI completely, and why?

**Common ideas:** secrets, customer data, payment, auth, security, private company code.
**Useful language:** "I avoid AI when...", "The cost of a mistake is too high", "Accountability stays with..."
**Pattern to apply:** High-risk area -> reason -> safer alternative.

**Version 1 (Clear B2):**
I avoid AI when the work involves sensitive data, secrets, customer information, or security-critical logic. I also avoid copying AI code directly into production without review. These areas need strong human judgment and local context. AI can help explain general ideas, but it should not make the final decision. Speed is not worth a privacy or security mistake.

**Version 2 (Expanded natural answer):**
I avoid AI when the task involves private user data, API keys, payment logic, or authentication decisions. Those areas are too sensitive, and the cost of a mistake is high. Even if AI gives a good-looking answer, I cannot trust it without deep review. I also avoid sharing company code if the tool is not approved by the team. The boundary is stricter when data or trust is involved.

For example, I would not paste raw customer logs, secrets, internal incidents, or private business data into an external AI tool. If AI is needed, I would anonymize the context or use an approved internal workflow. AI can help explain general ideas, but it should not become a shortcut around privacy and security rules. Speed is not worth a privacy or security incident.

### 3. What should be in a practical verification checklist?

**Common ideas:** requirements, edge cases, tests, security, accessibility, local patterns.
**Useful language:** "My checklist is simple", "Before using it, I check...", "Do I understand it well enough to own it?"
**Pattern to apply:** Requirement fit -> edge cases -> tests -> risk -> ownership.

**Version 1 (Clear B2):**
A practical verification checklist should start with the requirement. Then I check edge cases, tests, security risk, accessibility, and code review. For frontend work, I also check loading, empty, error, and success states. If AI generated the code, I ask whether it matches our project and whether I understand it well enough to own it. The checklist should be short enough to use every day.

**Version 2 (Expanded natural answer):**
My checklist is simple. First, does the output match the requirement? Second, did it handle edge cases, not only the happy path? Third, do tests pass? Fourth, does it follow our existing code style and patterns? For AI-generated code, I also ask myself: do I understand every part well enough to own it? If I cannot explain it, I should not merge it.

For AI-assisted work, I would add source checks and risk level. If AI mentions an API, I verify it against official docs or the local codebase. If the change touches auth, payment, privacy, or AI output quality, I require deeper review. Verification should match the risk, not just the size of the code. A small change in a critical flow can still need serious checking.

### 4. How do you keep accountability clear when AI contributes to a change?

**Common ideas:** ownership, PR transparency, verification evidence, reviewer focus.
**Useful language:** "AI helped draft...", "I verified it by...", "I still own what I merge."
**Pattern to apply:** AI contribution -> verification -> owner -> reviewer signal.

**Version 1 (Clear B2):**
Accountability should stay with the engineer who submits the change. If AI helped write code, tests, or documentation, I still need to understand and verify the final result. I would not say "AI wrote it" as an excuse. A clear rule is simple: if I merge it, I own it. That mindset keeps AI useful without weakening responsibility.

**Version 2 (Expanded natural answer):**
In a real pull request, I would be transparent about AI help when it matters. For example, I might say, "AI drafted the first test cases, and I verified them against the payment error scenarios." That helps reviewers know where to focus. It also shows that AI was part of the workflow, but not the final decision-maker. The engineer still owns what gets merged.

Accountability matters because AI can increase output faster than review capacity. I would classify AI-assisted changes by risk. Low-risk docs or simple tests can move faster, but auth, payment, privacy, and user-facing AI behavior need stronger review. The core principle stays the same: AI can contribute, but humans own safety, context, and final approval. "AI helped" should never become an excuse.

### 5. What are the benefits and limits of AI-generated code in team workflows?

**Common ideas:** boilerplate speed, first draft, system context, code review, ownership.
**Useful language:** "The benefit is...", "The limit is...", "I treat it like..."
**Pattern to apply:** Benefit -> where useful -> where risky -> review rule.

**Version 1 (Clear B2):**
AI-generated code can save time and reduce boilerplate, especially for small and clear tasks. It can help with utility functions, test drafts, or simple refactors. The limit is that it may not understand the whole system or the business rule. So the team still needs code review, tests, and ownership. I treat AI code as a draft, not as trusted code.

**Version 2 (Expanded natural answer):**
AI-generated code is helpful when the task is small and clear, like a utility function, test draft, or simple refactor. It can reduce boilerplate and help the team move faster. But it becomes risky when the task needs deep product context. Sometimes the code looks clean but does not match our business rule, data contract, accessibility requirement, or privacy boundary. That is why AI code should go through the same review process as human code.

I would treat AI-generated code like code from a new teammate. It may be useful, but it needs review, tests, and explanation before it becomes trusted. The benefit is faster first drafts, but the limit is context. If nobody understands why the code works, the speed gain is not real. The team still needs ownership.

### 6. How does AI usage differ for beginner versus experienced engineers?

**Common ideas:** shortcut vs thinking partner, prompt quality, review quality, skill growth.
**Useful language:** "The tool is the same, but...", "Experienced engineers...", "Beginners should..."
**Pattern to apply:** Beginner risk -> experienced behavior -> learning rule.

**Version 1 (Clear B2):**
Beginner engineers may use AI mainly to get answers quickly. Experienced engineers use AI to explore options, challenge ideas, and speed up drafts. The difference is review quality and ownership. Experienced engineers know when the answer is wrong, incomplete, or too generic. Beginners should learn to explain AI-generated code before they use it.

**Version 2 (Expanded natural answer):**
For beginners, AI can explain concepts and reduce fear, which is useful. But it can be dangerous if they copy code without understanding it. They may get something working without knowing the trade-offs or hidden bugs. Experienced engineers use AI differently: they ask better questions, give more context, and review the answer carefully. They also know when to ignore the suggestion.

The main difference is ownership. A beginner may use AI as a shortcut to get code working. An experienced engineer uses AI as a thinking partner but still owns the final decision. I think teams should teach juniors to explain AI-generated code before using it. That way, AI supports learning instead of replacing reasoning.

Quality should also win when AI-assisted work affects user-facing answers. Bad output can damage trust even if the system technically works. For low-risk tasks like documentation drafts or simple test ideas, speed is fine. For high-risk tasks involving user data, financial actions, identity, compliance, or long-term architecture, verification is the real priority. AI changes the speed of drafting, not the responsibility for safety.

### 8. You said AI improved speed. How do you prove quality did not drop?

**Common ideas:** defect rate, review rework, tests, CI failures, production bugs.
**Useful language:** "Speed is easy to feel, but quality needs evidence", "I would compare...", "The whole delivery cycle matters."
**Pattern to apply:** Speed claim -> quality metrics -> before/after -> conclusion.

**Version 1 (Clear B2):**
I would prove it with tests, review results, QA feedback, and production metrics. If we move faster but bugs increase, quality dropped. I would compare defect rate before and after using AI. I would also check whether reviewers spend more time fixing AI mistakes. Speed only counts if the whole delivery cycle improves.

**Version 2 (Expanded natural answer):**
Speed is easy to feel, but quality needs evidence. I would check post-merge bugs, QA feedback, test coverage, and review comments. If AI helps us write tests faster and escaped bugs stay the same or decrease, that is a good sign. But if reviewers spend more time fixing AI mistakes, the speed gain may be fake. We need to measure the whole delivery cycle, not only first-draft time.

I would track both process and outcome metrics. Process metrics include review rework, failed CI checks, and test coverage. Outcome metrics include production bugs, rollback rate, support tickets, and user complaints. AI may reduce drafting time but increase review time. So I would ask: are we delivering faster with the same or better quality, or are we just moving the cost to reviewers and QA?
